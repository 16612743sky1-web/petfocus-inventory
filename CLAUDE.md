# CLAUDE.md — 펫포커스 현장재고

이 파일은 이 저장소에서 작업하는 AI 어시스턴트(및 사람)를 위한 단일 기준 문서다.
여기 적힌 내용은 2026-07-12 기준 코드에서 직접 확인된 사실만 담았다.
`[HARD RULE]` 표시된 항목은 사용자 승인 없이 절대 어기지 않는다.

---

## 1. 프로젝트 개요

- **무엇**: 펫 간식(우피 츄잉 등) 자체 생산·판매 + 맘엔펫 제품 납품을 하는 2인(형/동생)이
  현장에서 스마트폰으로 쓰는 **재고 관리 웹앱**. UI 언어는 한국어.
- **화면**: 하단 탭 4개 — 재고 / 입출고 / 이력 / 원가.
- **사용 환경**: 모바일 우선. 레이아웃 최대 폭 520px(`.wrap`), viewport 확대 금지 설정.

## 2. 스택과 아키텍처

- **저장소 산출물은 `index.html` 단 하나** (HTML + CSS + JS 전부 인라인, 약 700줄).
  빌드 단계·패키지 매니저·프레임워크·번들러 없음.
- **호스팅**: GitHub Pages (정적 서빙). `main` 브랜치에 push하면 그대로 배포된다.
  별도 배포 스크립트나 CI 없음.
- **백엔드**: Firebase **Firestore** (JS SDK **v10.12.0**, gstatic CDN에서 dynamic `import()`).
  익명 인증(`signInAnonymously`)을 시도하며, 실패해도 계속 진행한다
  (보안 규칙이 `request.auth != null`을 요구한다는 가정의 주석이 코드에 있음).
- **오프라인/미설정 폴백**: `FIREBASE_CONFIG`가 비어 있거나 초기화 실패 시
  `localStorage`(키 `pf_state`) 단일 기기 모드로 동작하고 경고 배너(`#offBanner`)를 띄운다.
- **외부 리소스는 두 곳뿐**: Pretendard 폰트(jsdelivr CDN), Firebase SDK(gstatic CDN).

### [HARD RULE] 아키텍처 불변 조건
1. **단일 파일 유지** — 새 파일(js/css 분리, 설정 파일, 프레임워크, npm 등)을 만들지 않는다.
   문서(md) 추가도 사용자가 요청했을 때만.
2. **이중 스토리지 유지** — 데이터 읽기/쓰기 기능을 추가·수정할 때는 반드시
   Firestore 분기(`FS` truthy)와 localStorage 분기(`FS` null) **둘 다** 구현한다.
   한쪽만 고치고 끝내지 않는다.
3. **`FIREBASE_CONFIG` 블록(`▼▼▼` ~ `▲▲▲` 주석 사이)은 사용자 관리 영역** —
   값을 바꾸거나 지우지 않는다. 이 `apiKey`는 Firebase 웹앱의 공개 클라이언트 식별자로,
   비밀키가 아니다. "키가 노출됐다"며 삭제·마스킹·회전을 제안하거나 실행하지 말 것.
   실제 접근 제어는 Firestore 보안 규칙 + 익명 인증이 담당한다.
4. **재고 수량 변경은 원자적으로** — Firestore 모드에서 재고 증감은 반드시
   `writeBatch` + `FS.increment(delta)`를 쓴다 (`commitMove`, `removeLog` 참고).
   stock을 읽어서 계산 후 덮어쓰는(read-modify-write) 코드를 작성하지 않는다.
   두 사용자가 동시에 입력하는 상황이 이 앱의 핵심 시나리오다.
5. **Firebase SDK 버전(10.12.0)을 임의로 올리지 않는다.** 올려야 한다면
   app/firestore/auth 세 모듈을 같은 버전으로 함께 올리고 사용자에게 먼저 알린다.

## 3. 데이터 모델 (Firestore)

| 컬렉션 | 문서 ID | 필드 |
|---|---|---|
| `products` | 시드 배열 인덱스 문자열 `"0"`~`"8"` | `name`(문자열), `stock`(숫자), `order`(숫자, 정렬용), `min`(숫자, 선택 — 적정재고. 없으면 기본값 `LOW=20`) |
| `logs` | 자동 생성 ID | `pid`, `pname`, `type`(`'in'`\|`'out'`), `qty`, `chan`, `user`, `memo`, `ts`(`Date.now()` 밀리초), `date`(`"MM.DD"` 표시용 문자열) |

- `products`는 최초 접속 시 컬렉션이 비어 있으면 `SEED` 배열로 1회 시드된다.
- `logs` 실시간 구독은 `ts` 내림차순, `limit(300)`.
- localStorage 모드의 `pf_state`는 `{products, logs}` JSON이며 필드 구조는 위와 동일
  (로그 ID만 `'l'+Date.now()` 형식).
- 필드 추가·이름 변경은 **양쪽 모드와 기존 저장 데이터 호환**을 함께 고려해야 한다.
  기존 문서에 없는 필드는 `p.min!=null ? p.min : LOW`처럼 기본값 폴백으로 읽는 것이 이 코드의 관례.

## 4. 도메인 규칙 (코드에 박힌 비즈니스 로직)

- **분류**: 제품 이름이 `'맘엔펫'`으로 시작하면 `buy`(맘엔펫 납품/매입), 아니면 `made`(자체생산/주력).
  분류는 **이름 접두사로 판정**한다(`categoryOf`). 제품명을 바꿀 때 이 규칙이 깨지지 않는지 확인할 것.
- **단위**(`unitInfo`): 맘엔펫 제품은 `box`, 그 외는 `EA`.
  맘엔펫 중 이름에 `알래스카연어`가 들어가면 box↔ea 환산 없음(`per:0`), 나머지 맘엔펫은 `1box = 12ea`.
- **재고 상태**: `stock <= 0` → 품절(out), `0 < stock <= min` → 부족(low). 기본 적정재고 `LOW = 20`.
  제품별 `min`을 0으로 두면 알림 대상에서 제외된다.
- **판매처 채널**(출고 시): 네이버스토어 / 쿠팡 / 펫프렌즈 / 기타. 입고는 채널 대신 `'생산입고'` 고정.
- **담당자**: `형` / `동생` 두 명뿐. 기본값 `형`.
- **원가 탭**: 배치 단위 제조원가 계산기(재료비+인건비+전기). 마진은 부가세 제외 공급가 기준.
  이 탭의 단가/투입량은 저장되지 않고 입력할 때마다 즉석 계산된다(의도된 동작).
- **CSV 내보내기**: BOM(`﻿`) 포함 — 한글 엑셀 호환용. 제거하지 말 것.

## 5. 코드 관습

- **일반 `<script>`(모듈 아님) + 전역 함수 + 인라인 `onclick`** 스타일.
  새 핸들러도 이 방식을 따른다. `<script type="module">`로 바꾸면 인라인 핸들러가 전부 깨진다.
- DOM 접근은 `document.getElementById`, 렌더링은 `render()` 한 함수가 전체 화면을 다시 그린다.
  상태 변경 후에는 (Firestore 모드에선 `onSnapshot`이, 로컬 모드에선 직접 호출로) `render()`가 돌게 한다.
- **[HARD RULE] 사용자 입력 문자열을 `innerHTML`에 넣을 때는 반드시 `esc()`를 거친다**
  (제품명, 메모, 채널, 담당 등). 기존 코드가 이미 지키고 있는 XSS 방어선이다.
- CSS는 `:root` 커스텀 프로퍼티(`--brand`, `--out`, `--warn` 등) 기반의 브라운/크림 팔레트.
  새 색상은 하드코딩하지 말고 기존 변수를 재사용하거나 `:root`에 추가한다.
- 숫자 표시는 `tabular-nums` + `toLocaleString()`.
- 사용자에게 보이는 모든 문구·에러 메시지는 **한국어**.
- 비동기 저장 버튼 패턴: 누르면 `disabled` + "저장 중…" → `try/catch`로 실패 시
  한국어 alert/인라인 에러 → `finally`에서 복구. 새 저장 동작도 이 패턴을 따른다.

## 6. 완료의 정의 (Definition of Done)

작업은 아래 체크리스트를 **전부 통과**해야 완료다. 하나라도 확인 못 했으면
완료라고 보고하지 말고 "무엇을 확인 못 했는지"를 명시한다.

### 마무리 전 체크리스트
1. `git status`로 **변경 파일이 의도한 것뿐인지** 확인 (원칙적으로 `index.html` 하나).
2. 로컬에서 실행해 확인: `python3 -m http.server 8000` → `http://localhost:8000` 접속
   (Firebase dynamic import 때문에 `file://`로 열지 말 것).
3. 브라우저 콘솔에 **에러 0건** (익명 인증 미설정 경고 `console.warn`은 허용).
4. **탭 4개(재고/입출고/이력/원가) 모두 열어 렌더 확인.**
5. 입출고 흐름 검증: 입고 1건 저장 → 재고 탭 수치 증가 확인 → 이력에서 해당 기록
   삭제 → 재고가 원래 값으로 복귀하는지 확인.
6. **로컬 모드도 확인**: DevTools에서 네트워크를 offline으로 하거나 `FIREBASE_CONFIG.projectId`를
   임시로 `""`로 바꿔 새로고침 → 경고 배너 표시 + 기능 동작 확인 → **반드시 원복**.
7. 모바일 뷰포트(폭 ~390px)에서 레이아웃 깨짐 없는지 DevTools 기기 모드로 확인.
8. 새로 넣은 `innerHTML` 삽입 지점에 `esc()` 누락 없는지 재확인.
9. 재고 증감 코드가 batch+increment 패턴인지 재확인 (하드 룰 4).

### 커밋·배포
- 관측된 커밋 관습: 제목 한 줄 + 필요 시 한국어 본문 요약 (예: "재고부족 알림·동시수정·보안 개선").
  제목만 "Update index.html"로 두더라도 **본문에 무엇을 왜 바꿨는지 한국어로 한 줄** 남긴다.
- `main`에 push = 즉시 배포임을 항상 인지할 것. 확신 없는 변경은 브랜치에서 작업하고
  사용자에게 확인받는다.

## 7. 하지 말 것 (요약)

- 파일 분리, 프레임워크/빌드도구 도입, `type="module"` 전환 — [HARD RULE 1, §5]
- `FIREBASE_CONFIG` 수정·삭제, apiKey를 비밀로 취급해 지우기 — [HARD RULE 3]
- 재고 read-modify-write — [HARD RULE 4]
- Firestore 분기만/로컬 분기만 고치기 — [HARD RULE 2]
- `esc()` 없는 innerHTML 삽입 — [HARD RULE, §5]
- CSV의 BOM 제거, UI 영어화, 근거 없는 SDK 버전 업
- 이 문서에 없는 "선호"를 추측해서 적용하기 — 애매하면 작업을 멈추고 사용자에게 질문한다.

## 8. 알려진 한계 / 미확정 사항 (작업 시 참고)

- GitHub Pages 활성 여부·커스텀 도메인은 저장소 안에서 확인 불가(설정 화면 정보). 배포 관련
  작업 전에 사용자에게 확인.
- Firestore 보안 규칙의 실제 내용은 이 저장소에 없다(Firebase 콘솔 관리).
  규칙에 의존하는 변경(새 컬렉션 등)은 사용자에게 규칙 수정 필요 여부를 알릴 것.
- 자동화된 테스트 없음 — §6 수동 체크리스트가 유일한 검증 수단.
- `removeLog`는 로그 전체를 `JSON.stringify`해 onclick에 인라인으로 넘긴다. 로그 필드를
  추가하면 이 경로(따옴표 이스케이프 포함)도 함께 검증할 것.
