# HTML 공유 안전 처리 (html-share)

## 왜 필요한가
직접 만든 HTML을 파일로 전달하면, `<meta charset="utf-8">` 선언이 없을 때 브라우저가 인코딩을
스스로 추측한다. 데스크톱 크롬은 대충 맞히지만 **iOS Safari·일부 모바일은 한글을 엉뚱한 인코딩으로
읽어 깨진 글자(泥·援·猿 같은 모지바케)로** 보여준다. **영문·숫자는 멀쩡하고 한글만 깨지는** 증상이면
거의 항상 이 원인이다.

특히 Claude가 만든 상세페이지·목업 조각 HTML(`<title>`+`<style>`+`<div>`로 시작, `<head>`가 없는
형태)은 charset이 빠지기 쉽다. 아티팩트로 볼 땐 플랫폼이 charset을 자동으로 붙여 멀쩡하지만,
**파일로 내려받아 열면 깨진다.** 그래서 파일로 전달하기 전에 이 처리를 반드시 거친다.

## 언제 적용하나
- 사용자에게 **HTML을 파일로 전달**하기 직전 (SendUserFile 등으로 보내기 전)
- 사용자가 "다른 PC/폰에서 열려야 해", "팀에 공유할 거야"라고 할 때
- "한글이 깨진다"는 스크린샷·제보가 왔을 때 (원인 진단 + 수정)

아티팩트(claude.ai)로만 볼 거라면 필수는 아니다(플랫폼이 charset을 붙임). 하지만 **파일 공유가
섞일 가능성이 조금이라도 있으면 그냥 적용**하는 게 안전하다 — 부작용이 없고 멱등이다.

## 어떻게 하나
번들 스크립트를 쓴다. 손으로 편집하지 말 것 — 스크립트가 조각/완전문서/이미-charset있음을 모두
알아서 처리하고, 중복 삽입도 막는다(멱등).

```bash
python3 <이-스킬-경로>/scripts/make_shareable_html.py <입력.html> [출력.html]
```

- 출력 경로를 생략하면 `<이름>_공유용.html`로 저장하고 최종 경로를 stdout에 출력한다.
- 스크립트가 하는 일:
  1. `<head>`가 있으면 → 없는 `charset`·`viewport` 메타만 보충
  2. 조각(head/html/body 없음)이면 → `<!doctype html><html lang="ko"><head>…</head><body>…</body></html>`로 감싸기
  3. 이미 charset이 있으면 → 그대로 두고 중복 안 만듦
  4. UTF-8(BOM 없음)로 저장 (BOM은 불필요하고 간혹 첫 글자를 깨뜨림)

## 절차
1. 전달할 HTML에 스크립트를 돌려 `_공유용.html`을 만든다.
2. 스크립트 stdout의 경로를 확인한다. (경고가 뜨면 charset이 앞부분에 안 잡힌 것 —
   그 파일을 열어 `<head>` 안 첫 줄에 `<meta charset="utf-8">`가 있는지 눈으로 확인)
3. **가공된 `_공유용.html`을 사용자에게 전달**한다(원본 말고).
4. 전달 메시지에 한 줄 덧붙이면 친절하다: "charset을 넣어 아이폰·안드로이드·PC 모두 한글이
   정상 표시됩니다."

## 진단 팁 (사용자가 "깨진다"고 할 때)
- **한글만 깨지고 영문·숫자는 멀쩡** → charset 문제. 이 스킬로 해결.
- **전부 네모/물음표** → 폰트 문제(글꼴 없음)일 수 있음. charset과 별개.
- 어떤 기기·앱으로 열었는지 물어보면 원인 좁히기 쉽다(iOS Safari가 이 문제에 특히 취약).

## 예방 (새로 HTML을 만들 때)
처음부터 문서 앞에 아래를 넣으면 이 처리가 애초에 필요 없다. 새 HTML을 작성할 땐 습관적으로 넣자.
```html
<!doctype html>
<html lang="ko">
<head>
<meta charset="utf-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
```
단, **아티팩트로 발행할 파일**은 플랫폼이 head 골격을 자동으로 감싸므로 doctype/html/head/body를
직접 넣지 않는다(그 경우엔 조각으로 두고, 파일 공유 시에만 이 스킬로 감싼다).
